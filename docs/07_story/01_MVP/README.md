## Story 文档结构（统一模板）

每个 Story 文档含：

1. **元信息头**：阶段 / 类型 / AC / 状态 / 上游链接
2. **用户故事**：As a / I want / so that
3. **验收标准（AC）**：编号 + 验收项 + Pass 判据（黑盒可验）
4. **技术设计要点**：场景与节点 / 脚本契约 / 关键决策（ADR 引用）/ 可视化搭建协作（§12.4 标注）
5. **依赖**：前置 / 架构依据 / 被依赖
6. **测试策略**：单元/集成/功能测试覆盖点
7. **验证步骤（§12.5）**：运行方式 + 逐 AC 操作 + 预期表现 + 异常判定

## 开发顺序与并行性

Story 严格按 M1→M26 顺序开发（每个里程碑 ≤10min 净开发量，架构 §15.1）。部分阶段内部存在可并行性，但 MVP 建议串行以保持小步验证：

- **强串行链**：M1→M2（重力跳跃）→M5（状态机）→M7（攻击）→M8（命中）→M11（追击）→M12（受击）→M13（击杀）
- **可独立分支**：M3/M4（填皮）、M9/M10（野猪占位填皮）、M17（视差）、M25（shader）可相对独立插入

> 每个 Story 完成后走宪法 §12.7 收尾闭环：经验沉淀 → 提交经验文档 → 飞书通知。
